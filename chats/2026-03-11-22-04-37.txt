`82400` 这个 WebContent 进程里，最大的有效桶是：
- `WebKit Malloc` `287.2 MB resident`，其中 `118.8 MB dirty`
- `DefaultMallocZone` `13.2 MB resident`
- `JS JIT generated code` 只有 `5.1 MB resident`

这说明大头不是 JIT 代码，也不是 Go 对象，而是 WebKit 自己的页面堆内存: DOM、JS 对象、布局缓存、解码后的页面资源、WebKit/bmalloc 缓存这一类。

3. GPU 进程现在不是主因  
GPU 进程里最吓人的一项是：
- `IOSurface 303.1 MB`
但它是 `0K resident`，而且 `271.5 MB empty`

这类数字更多是图形表面池和保留映射，不代表当前真的占了那么多物理内存。它当前真实 footprint 只有 `32.3 MB`。所以你截图里那个 “Graphics and Media 100+ MB” 更像 Activity Monitor 的映射/驻留视角，不是当前独占压力。

所以这次定位可以落成一句话：

当前最可疑的不是 Go 后端泄漏，也不是 GPU 常驻爆炸，而是 `WKWebView` 的 `WebContent` 进程把页面访问后的对象和 WebKit 内部缓存留在了 `WebKit Malloc` 里。

还有两个很重要的附带结论：
- `JS VM Gigacage` 那种几十 GB 是保留地址空间，不是实际内存。
- Activity Monitor 的“内存”数字比 `Physical footprint` 更容易把共享映射和保留区也算得很吓人，所以它比真实压力更悲观。

这轮我没改任何代码，工作区是干净的。

单入口、隐藏双角色  
用户永远只启动 `SkillFlow.app` / `SkillFlow.exe`。内部由同一个可执行文件自举成两个隐藏角色：
- `helper`：常驻，负责托盘/菜单栏、单实例、后台服务、拉起/聚焦 UI
- `ui`：Wails 窗口进程，负责界面；关闭窗口后直接退出，连带 WebKit 进程一起退出

这个“角色”只能在内部存在，通常用隐藏参数或环境变量区分。用户不会看到，也不需要理解。没有这个内部区分，操作系统层面就没法知道当前进程到底该当 helper 还是 UI。

结合当前代码的最小切口
我看过现在的结构，最小改法是：

1. 启动入口改成 bootstrap  
当前 [main.go](/Users/shinerio/Workspace/code/SkillFlow/cmd/skillflow/main.go) 一上来就 `ensureSingleInstance()` 然后直接 `wails.Run(...)`。

- 先执行 `bootstrapLaunch()`
- 如果当前进程应该成为 `helper`，就不进 Wails 主窗口
- 如果当前进程应该成为 `ui`，才进入 `wails.Run(...)`

2. 单实例从“窗口实例”改成“helper 实例”  
现在只有 Windows 有单实例，而且它默认“已有实例就是能直接 show 的窗口”，见 [single_instance_windows.go](/Users/shinerio/Workspace/code/SkillFlow/cmd/skillflow/single_instance_windows.go)。  
这要改成：
- helper 才是唯一主实例
- 第二次双击应用时，不是唤醒当前窗口，而是给 helper 发 `show-ui` 命令然后退出

3. 托盘/菜单栏从“直接控窗口”改成“控 UI 进程”  
现在 Windows 托盘菜单直接调用 `showMainWindow()` / `quitApp()`，[tray_windows.go](/Users/shinerio/Workspace/code/SkillFlow/cmd/skillflow/tray_windows.go#L400)。  
macOS 菜单栏也是同样模式，[tray_darwin_callbacks.go](/Users/shinerio/Workspace/code/SkillFlow/cmd/skillflow/tray_darwin_callbacks.go#L9)。  
最小改法是把这层抽成：
- `Show` -> `helper.ensureUIRunningAndFocused()`
- `Hide` -> 对 UI 发隐藏/退出命令
- `Quit` -> helper 杀 UI，再自己退出

4. UI 关闭时不再隐藏  
当前窗口关闭仍是隐藏，见 [main.go](/Users/shinerio/Workspace/code/SkillFlow/cmd/skillflow/main.go#L27)。  
双进程后应该改成：
- `HideWindowOnClose: false`
- UI 关闭按钮直接退出 UI
- helper 继续活着，托盘/菜单栏仍可重新打开

最小内部通信
第一版不需要重 RPC 框架，只要一个本机控制通道：
- 命令：`show-ui`、`focus-ui`、`quit-all`
- Unix 用 domain socket
- Windows 用 named pipe

这足够先把“关闭窗口释放 WebKit，但托盘还能重开”做出来。

最关键的结构判断
如果你要“保留 Go 后端、退出 WebKit”，那业务真值最终必须落到 helper，不应继续绑在 Wails `App` 上。否则 UI 一退出，后端也一起没了。  
所以真正稳的演进顺序是：

1. 先做 `bootstrap + helper 单实例 + tray/menu 控制 UI`
2. 再把当前 `App` 里的业务方法逐步抽到一个 headless service
3. UI 改成只通过 IPC 调 service

这比“在单进程里强行热销毁并重建 WebView”稳得多。
