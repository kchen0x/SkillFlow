  架构方案对比

  核心问题是：如何设计可扩展的插件体系（云厂商、工具适配器、安装方式），以及异步
  操作的通知机制。

  ---
  方案 A：纯接口注册（Interface Registry）

  所有扩展点定义为 Go interface，具体实现编译时注册进来。无动态加载，简单纯粹。

  core/
    interfaces.go     # CloudProvider, ToolAdapter, Installer 接口
    registry.go       # 注册表，编译时注入实现
  providers/
    aliyun/, tencent/, huawei/
  adapters/
    claude-code/, opencode/, codex/

  - 优点：Go 惯用，测试友好，无反射魔法
  - 缺点：异步操作（备份进度、同步状态）需要轮询或回调，稍显笨拙

  ---
  方案 B：事件总线驱动（Event Bus）

  核心操作通过事件总线驱动，UI
  订阅事件。所有异步操作（备份、同步、更新检测）都通过事件推送状态。

  core/
    eventbus.go       # 中心事件总线
    services/         # 各业务 service 发布事件
  ui/
    handlers/         # 订阅事件，更新前端

  - 优点：实时反馈体验好，解耦彻底
  - 缺点：事件流追踪复杂，调试难度高，对小型项目略显过度设计

  ---
  方案 C（推荐）：接口扩展 + Channel 异步通知

  业务逻辑用 interface 定义扩展点，异步操作用 Go channel 传递进度/状态，Wails 的
   EventsEmit 将 channel 数据推送到前端。

  core/
    skill/            # skill 核心模型与存储
    sync/             # 同步适配器接口 + 实现
    backup/           # 云备份接口 + 实现
    install/          # 安装方式接口 + 实现
    notify/           # channel-based 通知中心
  app/
    wails/            # Wails App，调用 core，监听 channel 转发事件到前端
  frontend/           # React + TypeScript UI

  - 优点：简单接口保证可扩展性，channel 保证异步操作有实时反馈，架构分层清晰
  - 缺点：需要合理设计 channel 生命周期，避免泄漏

  ---
  推荐方案 C，原因：Go channel 是语言原生异步机制，与 Wails EventsEmit
  天然配合；interface 注册模式是最 Go-idiomatic
  的扩展方式；整体复杂度适中，符合项目当前规模。

  你认可方案 C 吗？还是有其他偏好？

  安装接口

  // core/install/installer.go

  type InstallSource struct {
      Type string // "github" | "local"
      URI  string // GitHub repo URL 或本地路径
  }

  type SkillCandidate struct {
      Name      string
      Path      string // 在源中的相对路径
      Installed bool   // 是否已安装（用于 UI 勾选状态提示）
  }

  type Installer interface {
      // 扫描源，列出所有可安装的 skill 候选
      Scan(ctx context.Context, source InstallSource) ([]SkillCandidate, error)
      // 安装选中的 skills，通过 notify channel 报告进度
      Install(ctx context.Context, source InstallSource, selected
  []SkillCandidate, category string) error
      // 标识符，用于注册表匹配
      Type() string
  }

  同步适配器接口

  // core/sync/adapter.go

  type SyncDirection string

  const (
      SyncTo   SyncDirection = "to"   // SkillFlow -> 工具
      SyncFrom SyncDirection = "from" // 工具 -> SkillFlow
  )

  type ToolAdapter interface {
      Name() string        // "claude-code" | "opencode" | ...
      DefaultSkillsDir() string  // 各工具的默认 skills 目录
      // 推送 skills 到工具目录（拍平，不带分类结构）
      Push(ctx context.Context, skills []Skill, targetDir string) error
      // 从工具目录拉取 skills
      Pull(ctx context.Context, sourceDir string) ([]Skill, error)
  }

  // core/backup/provider.go

  type CloudProvider interface {
      Name() string  // "aliyun" | "tencent" | "huawei"

      Init(credentials map[string]string) error
      // 上传整个 skills 目录（压缩打包后上传）
      Upload(ctx context.Context, localDir string, bucket string) error
      // 下载并恢复
      Download(ctx context.Context, bucket string, targetDir string) error
      // 列出备份记录
      ListBackups(ctx context.Context, bucket string) ([]BackupRecord, error)
      // 返回用户需要填写的凭证字段（用于动态渲染配置 UI）

  // core/backup/provider.go

  type CloudProvider interface {
      Name() string
      Init(credentials map[string]string) error

      // 同步本地目录到云端（增量，保持目录结构一致）
      Sync(ctx context.Context, localDir string, bucket string, remotePath
  string) error

  string) error

      List(ctx context.Context, bucket string, remotePath string) ([]RemoteFile,

  }

          │

          │
          ▼
  触发自动云备份

  GitHub 更新检测流程

  // core/update/checker.go

  type UpdateResult struct {
      SkillID     string
      HasUpdate   bool
      LatestSHA   string  // 远端最新 commit SHA（skills 子目录）
      CurrentSHA  string  // 本地记录的 SHA
  }

  - 安装时记录该 skill 目录对应的最新 commit SHA（通过 GET 
  /repos/{owner}/{repo}/commits?path=skills/skill-a&per_page=1 获取）
  - 更新检测时重新请求该接口，对比 SHA
  - 检测时机：
    - 应用启动时后台静默检测一次
    - 用户在 Dashboard 手动点击"检查更新"
    - 检测到更新后在 SkillCard 上显示 🔴 角标 + 通知
  - 更新操作：重新下载覆盖本地文件，更新 meta 中的 SHA 和 updatedAt
